------------------Docker build----------------------------------------------------------------
`docker build -t <image_name> . `→ image created → `docker ps` → 
`docker run -d -p <host_port>:<container_port> <image_name> `→ container starts → 
    application runs inside container → exposed on localhost:<host_port> → END

* `docker build -t <image_name> .` → Creates a Docker image from the Dockerfile in the current directory
* Image → A packaged snapshot of application + dependencies
* `docker ps` → Shows all running containers
* `docker run -d -p <host_port>:<container_port> <image_name>` → Starts a container and maps ports
* Container → A running instance of a Docker image
* Container starts → Docker initializes and runs the app inside container
* Application runs inside container → Your code executes in isolated environment
* `<container_port>` → Internal port where app listens inside container
* `<host_port>` → External port on your machine for access
* `localhost:<host_port>` → URL to access the running app in browser
* END → App is successfully running and accessible

-----------volume------------------------------
*docker volume create <volume_name> → creates a persistent storage area managed by Docker
*docker run -v <volume_name>:/app <image_name> → attaches volume to container for data storage
  Volume → a persistent storage layer that exists outside the container lifecycle
  Bind Mount → maps a host folder directly into the container filesystem
  -v <volume_name>:<container_path> → connects Docker volume to a folder inside container
  Data persistence → data is not lost even if container stops or is deleted
  Shared storage → multiple containers can use the same volume
*docker volume ls → lists all Docker volumes
*docker volume inspect <volume_name> → shows details of a specific volume
*END → data is safely stored outside container and survives restarts/deletion just list not list like blocks

*FROM – base image (starting environment) → FROM python:3-slim
*WORKDIR – sets working directory inside container → WORKDIR /app
  -WORKDIR name can be anything — it only defines the container’s working folder, not your system folder.
*COPY – copies files from host to container → COPY . /app
*RUN – executes commands during image build → RUN pip install -r requirements.txt
*EXPOSE – declares which port the app uses → EXPOSE 3000
*CMD – default command when container starts → CMD ["python", "index.py"]
*ENTRYPOINT – main command that always runs → ENTRYPOINT ["python"]
*ENV – sets environment variables inside container → ENV PYTHONUNBUFFERED=1
*USER – runs container as a non-root user → USER appuser

## Containerizing a Python Application with Docker: A Step-by-Step Approach

To containerize a Python application defined in `index.py`, I would begin by creating a `Dockerfile` in the same 
directory as the application code. This file serves as a blueprint for building a Docker image that encapsulates the
 application and its dependencies, ensuring consistent execution across any environment.

**Selecting a Base Image**  
First, I would specify a base image using the `FROM` instruction. A practical choice would be an official Python 
image such as `python:3.11-slim`. This provides a minimal Linux environment with Python pre-installed, eliminating 
the need to manually install the interpreter while keeping the final image size reasonable.
“Docker uses a Linux base image because containers share the host OS kernel, and Docker itself relies on Linux 
kernel features like namespaces and cgroups for isolation. A minimal Linux user-space provides the necessary system
 libraries and environment for applications to run, while keeping the image lightweight and portable. Since most production servers run Linux, using a Linux base ensures consistency and compatibility across environments.”

**Defining the Working Directory**  
Next, I would set a working directory inside the container using `WORKDIR /app`. All subsequent commands — copying
 files, installing dependencies, and running the application — would then execute relative to this directory. 
 This
  practice centralizes file operations and avoids path-related confusion.

**Copying Files Strategically**  
To optimize the build process, I would copy only the dependency manifest first — `requirements.txt` — using `COPY
 requirements.txt .`. This ordering is intentional: Docker caches each instruction as a layer. By copying 
 dependencies before the rest of the source code, any future changes to `index.py` or other application files 
 will 
 not invalidate the cached dependency installation layer, significantly speeding up repeated builds.

**Installing Dependencies**  
With `requirements.txt` inside the container, I would run `RUN pip install --no-cache-dir -r requirements.txt`.
 The `--no-cache-dir` flag prevents pip from storing unnecessary cache files, reducing the final image size. 
 At
  this point, all Python libraries required by the application are present inside the image.

**Copying the Application Code**  
After dependencies are installed, I would copy the remaining application code using `COPY . .`. This includes 
`index.py` and any other supporting files. Because this step occurs after the dependency installation, modifying 
the source code does not force Docker to reinstall packages — only this final layer is rebuilt.

**Exposing the Network Port**  
To allow external access to the application, I would document the expected network port with `EXPOSE 3000`.
 While this instruction does not actually publish the port, it serves as clear documentation for anyone running 
 the container and enables the `-P` flag to automatically map exposed ports.

**Defining the Startup Command**  
Finally, I would specify how to start the application using the `CMD` instruction. Using the **exec form** — 
`CMD ["python", "index.py"]` — rather than the shell form ensures that Python receives OS signals directly
 (e.g., `SIGTERM` for graceful shutdown) and that the process runs as PID 1 inside the container.

**Building and Running the Container**  
With the `Dockerfile` complete, I would open a terminal in the same directory and build the image using:  
`docker build -t my-python-app .`  
The `-t` flag assigns a name to the image, and the trailing dot indicates the build context is the current directory.

To run the container, I would execute:  
`docker run -d -p 3000:3000 my-python-app`  
- `-d` runs the container in detached mode (background).  
- `-p 3000:3000` maps port 3000 from the container to port 3000 on the host machine.  
After this command, the application would be accessible at `http://localhost:3000`.
